## Спочатку — три інструменти, які вирішують 80% задач

Перш ніж задачі, познайомся з трьома структурами. Майже всі рішення — це вони.

**HashMap (хеш-мапа)** — це як шафа з підписаними шухлядами. Ти кажеш "дай мені шухляду 'Anna'" — і одразу її отримуєш, не перебираючи всі. Доступ майже миттєвий, O(1). Використовуй, коли треба **щось запам'ятати під ключем**: рахувати, групувати, шукати пару.

**Set (множина)** — це як список гостей на вечірці. Тобі важливо лише одне: людина є чи нема. Дублікатів не буває. Перевірка "чи є X?" — миттєва.

**Stack (стек)** — це як стопка тарілок. Кладеш зверху, береш зверху. Остання покладена — перша взята (LIFO). Ідеально для дужок, бо остання відкрита дужка має закритися першою.

Якщо запам'ятаєш, **коли** тягнутися до кожного — половина роботи зроблена.

## Задача 1 — Середні бали

**Що хочуть:** дано картки `"Ім'я,Предмет,Бал"`. Порахувати середній бал кожного студента, потім назвати найкращого.

**Інтуїція:** уяви стос карток. Щоб порахувати середнє для людини, тобі треба дві речі: **сума всіх її балів** і **скільки в неї оцінок**. Середнє = сума ÷ кількість.

**Чому HashMap:** ти не знаєш наперед, скільки студентів і як їх звати. Тобі потрібна шафа, де на кожне ім'я є шухляда з двома числами. Тут шухляда — це `int[2]`, де `[0]` = сума, `[1]` = кількість.

**Покроково:**
1. Береш картку, розрізаєш по комі → ім'я і бал.
2. Знаходиш шухляду цього імені (або створюєш порожню `[0,0]`).
3. Додаєш бал до `[0]` і `+1` до `[1]`.
4. Коли картки скінчилися — у кожній шухляді ділиш суму на кількість.

**Трасування** на `Anna 90`, `Anna 80`, `Bob 70`:
```
Картка Anna,90 → шухляда Anna = [90, 1]
Картка Anna,80 → шухляда Anna = [170, 2]
Картка Bob,70  → шухляда Bob  = [70, 1]
Ділимо: Anna = 170/2 = 85.0,  Bob = 70/1 = 70.0
Найкращий: Anna (85 > 70)
```

**Ключова ідея:** один прохід по даних + миттєвий доступ за іменем = швидко й просто. Якби ти щоразу шукав студента перебором усього списку — було б повільно (O(n²)).

## Задача 2 — Баланси рахунків + овердрафт

**Що хочуть:** транзакції `"рахунок,тип,сума"` (DEBIT = списання, CREDIT = поповнення). Частина 1: фінальний баланс кожного рахунку. Частина 2: які рахунки **хоч раз пішли в мінус** (порядок транзакцій важливий!).

**Інтуїція до Частини 1:** це як банківська виписка. CREDIT додає гроші (+), DEBIT забирає (−). Ти просто йдеш по списку й додаєш/віднімаєш.

**Найважливіша ідея (Частина 2):** є різниця між **фінальним** балансом і **проміжним**.
Уяви: рахунок мав 0, зняли 100 (−100, мінус!), потім поклали 150 (+50). Фінал — плюс 50. Але **в середині** він провалився в −100. Якщо дивитися тільки на фінал — ти цього не побачиш. Тому треба перевіряти баланс **після кожної** транзакції, а не лише в кінці.

**Чому два окремі методи / дві мапи:** у Частині 1 нам байдужий порядок — результат однаковий. У Частині 2 порядок критичний, і ми ведемо "поточний" (running) баланс і ловимо момент, коли він стає від'ємним.

**Трасування** Частини 2 на `A DEBIT 100`, `A CREDIT 150`, `B DEBIT 50`:
```
A DEBIT 100  → баланс A = -100  → МІНУС! записуємо A
A CREDIT 150 → баланс A = +50   → плюс, але A вже у списку
B DEBIT 50   → баланс B = -50   → МІНУС! записуємо B
Результат: [A, B]
```

**Чому `long`, а не `double` (скажи це вголос!):** гроші не можна тримати в `double`, бо `0.1 + 0.2` у комп'ютері дає `0.30000000000000004`. У фінансах це недопустимо. Тому суми тримають у **цілих** — у центах. 50 євро = 5000 центів. Жодних дробів, жодних помилок округлення.

## Задача 3 — Пара з заданою сумою

**Що хочуть:** дано числа й `target`. Знайти два числа, що в сумі дають target, повернути їхні позиції.

**Наївний спосіб:** взяти кожне число й порівняти з усіма іншими. Два вкладені цикли — повільно, O(n²).

**Розумна ідея:** для кожного числа `x` тобі потрібен партнер `target − x`. Питання лише: **чи ми вже бачили цього партнера?** А "чи я вже бачив X?" — це робота для HashMap (миттєва перевірка).

**Аналогія:** ти на вечірці й шукаєш пару, чий зріст разом із твоїм = 9. Твій зріст 4 → шукаєш людину зростом 5. Замість обходити всіх щоразу, ти ведеш список "кого я вже зустрів". Зустрівши нового, одразу перевіряєш, чи потрібний партнер уже в списку.

**Покроково:**
1. Береш число `x` на позиції `i`.
2. Рахуєш, кого треба: `need = target − x`.
3. Дивишся в мапу: чи `need` там уже є? Якщо так — знайшли пару, повертаємо позиції.
4. Якщо ні — кладемо `x` у мапу (запам'ятали) і йдемо далі.

**Трасування** на `[2,7,11,15]`, target=9:
```
i=0, x=2  → need=7. Мапа порожня. Кладемо {2→0}
i=1, x=7  → need=2. Мапа має 2! → повертаємо [0, 1]
```

**Ключова ідея:** замінюємо "шукати партнера перебором" на "запам'ятати побачене й перевіряти миттєво". O(n²) → O(n).

## Задача 4 — Валідні дужки

**Що хочуть:** рядок із `()[]{}`. Перевірити, що дужки правильно вкладені й закриті. `({[]})` — ок, `(]` — ні.

**Інтуїція:** остання відкрита дужка має закритися **першою**. `( [ ] )` — спершу закривається `[`, потім `(`. Це точно поведінка **стопки тарілок** (стек): хто покладений останнім, той знімається першим.

**Покроково:**
1. Йдеш по символах.
2. Бачиш **відкриваючу** (`(`, `[`, `{`) — кладеш на стек.
3. Бачиш **закриваючу** — дивишся, що на вершині стека. Воно має бути **парною** відкриваючою. Якщо так — знімаєш зі стека (пара знайдена). Якщо ні (або стек порожній) — рядок невалідний.
4. У кінці стек має бути **порожній**. Якщо щось лишилося — є незакриті дужки.

**Трасування** на `({[]})`:
```
(  → кладемо, стек: [ (
{  → кладемо, стек: [ ( {
[  → кладемо, стек: [ ( { [
]  → закриваюча, вершина [ — пара! знімаємо, стек: [ ( {
}  → закриваюча, вершина { — пара! знімаємо, стек: [ (
)  → закриваюча, вершина ( — пара! знімаємо, стек: пусто
Кінець: стек порожній → ВАЛІДНО
```
А на `(]`:
```
(  → кладемо, стек: [ (
]  → закриваюча, вершина ( — НЕ пара для ] → НЕВАЛІДНО
```

**Ключова ідея:** стек природно відстежує "яку дужку треба закрити наступною".

## Задача 5 — Об'єднати інтервали

**Що хочуть:** дано відрізки `[[1,3],[2,6],[8,10]]`. Об'єднати ті, що перетинаються → `[[1,6],[8,10]]`.

**Інтуїція:** уяви зустрічі в календарі. Якщо одна йде 1:00–3:00, інша 2:00–6:00, вони накладаються → це насправді один блок 1:00–6:00.

**Чому спершу сортуємо:** якщо відрізки впереміш, важко зрозуміти, що з чим перетинається. Але якщо вибудувати їх **за початком** (зліва направо), то перетини стають сусідами — їх легко зливати по черзі.

**Як зрозуміти, що два відрізки перетинаються:** наступний починається **раніше або там само**, де закінчується поточний. Тобто `наступний.початок ≤ поточний.кінець`.

**Покроково:**
1. Сортуєш відрізки за початком.
2. Береш перший як "поточний".
3. Дивишся наступний: перетинається? → розширюєш кінець поточного до більшого з двох.
4. Не перетинається? → закриваєш поточний (додаєш у результат), наступний стає новим поточним.

**Трасування** на `[[1,3],[2,6],[8,10]]` (уже відсортовано):
```
поточний = [1,3]
[2,6]: 2 ≤ 3? так → перетин. Кінець = max(3,6) = 6. поточний = [1,6]
[8,10]: 8 ≤ 6? ні → нема перетину. Закриваємо [1,6]. поточний = [8,10]
Кінець: додаємо [8,10]
Результат: [[1,6],[8,10]]
```

**Ключова ідея:** сортування перетворює хаос на впорядкований ряд, де перетини — сусіди.

## Задача 6 — Топ користувачів із логів

**Що хочуть:** лог-рядки `"користувач дія"`. Повернути користувачів за спаданням активності; при однаковій кількості — за алфавітом.

**Інтуїція:** дві частини. Спершу **порахувати**, скільки разів кожен з'являється (це знову HashMap — рахуємо за ключем). Потім **відсортувати** за цією кількістю.

**Покроково:**
1. Для кожного рядка береш ім'я (перше слово) і додаєш +1 у мапу-лічильник.
2. Тепер є мапа `ім'я → кількість`. Сортуєш її записи: спочатку за кількістю (більше — вище), а якщо кількість однакова — за іменем за алфавітом.
3. Лишаєш тільки імена.

**Трасування** на `u1, u2, u1, u1, u2`:
```
Рахуємо: u1 → 3, u2 → 2
Сортуємо: u1 (3) вище за u2 (2)
Результат: [u1, u2]
```

**Чому нічию вирішуємо за алфавітом:** інакше порядок однакових був би непередбачуваний. Інтерв'юери люблять чіткий, детермінований результат. **Назви це сам** як edge-case.

## Задача 7 — Найдовший підрядок без повторів

**Що хочуть:** знайти довжину найдовшого шматка рядка, де **немає** повторюваних символів. `"abcabcbb"` → 3 (це `"abc"`).

**Інтуїція — "вікно", що повзе (sliding window):** уяви рамку, що виділяє шматок рядка. Ти рухаєш правий край рамки вправо, додаючи символи. Поки символи унікальні — вікно росте. Як тільки натрапив на повтор — підтягуєш **лівий** край рамки так, щоб повтор зник.

**Як ловимо повтор:** запам'ятовуємо, де востаннє бачили кожен символ (HashMap `символ → індекс`). Якщо символ уже зустрічався **всередині поточного вікна**, зсуваємо лівий край (`start`) одразу за те старе входження.

**Покроково:**
1. Рухаєш правий край `i` вправо.
2. Якщо символ уже був і його позиція ≥ `start` (тобто в межах вікна) → `start` = стара позиція + 1.
3. Оновлюєш останню позицію символа.
4. Поточна довжина вікна = `i − start + 1`. Запам'ятовуєш максимум.

**Трасування** на `"abcabcbb"` (це реальний вивід коду):
```
i=0 c=a start=0 вікно=[0..0] best=1
i=1 c=b start=0 вікно=[0..1] best=2
i=2 c=c start=0 вікно=[0..2] best=3   ← "abc", максимум
i=3 c=a start=1 вікно=[1..3] best=3   ← 'a' повторилась, ліву межу зсунули
i=4 c=b start=2 вікно=[2..4] best=3
i=5 c=c start=3 вікно=[3..5] best=3
i=6 c=b start=5 вікно=[5..6] best=3
i=7 c=b start=7 вікно=[7..7] best=3
Відповідь: 3
```

**Ключова ідея:** замість перевіряти всі можливі підрядки (повільно), ми одним проходом "тягнемо" вікно й лише підправляємо ліву межу. O(n).

## Задача 8 — Badge access (підписна задача Karat)

**Що хочуть:** записи `[ім'я, "enter"/"exit"]` — хто заходив і виходив із будівлі. Знайти порушення: 1) хто **вийшов, не зайшовши**; 2) хто **зайшов і не вийшов** (або зайшов двічі поспіль, не вийшовши).

**Інтуїція:** уяви охоронця, що тримає список "хто зараз усередині". Кожен запис оновлює цей список. Порушення — це коли запис не має сенсу: вихід людини, якої немає всередині, або повторний вхід того, хто вже всередині.

**Чому Set "inside":** нам важливо лише одне про кожну людину — вона зараз у будівлі чи ні. Це ідеальна робота для множини (є/нема).

**Покроково:**
1. На `enter`: якщо людина **вже** у множині `inside` — це порушення (зайшов двічі, не вийшовши) → записуємо в "зайшов без виходу". У будь-якому разі додаємо в `inside`.
2. На `exit`: якщо людини **немає** в `inside` — порушення (вийшов без входу) → записуємо. Інакше — прибираємо з `inside`.
3. Коли записи скінчилися — усі, хто **лишився** в `inside`, теж порушники ("зайшов і не вийшов").

**Трасування** на `Paul enter`, `Paul exit`, `Curtis enter`, `Mary exit`:
```
Paul enter   → inside = {Paul}
Paul exit    → Paul є всередині → прибираємо. inside = {}
Curtis enter → inside = {Curtis}
Mary exit    → Mary НЕМАЄ всередині → порушення "вийшов без входу" = [Mary]
Кінець: у inside лишився Curtis → "зайшов без виходу" = [Curtis]
```

**Чому це класика Karat:** реалістична, багатоетапна, з купою edge-cases (подвійний вхід, подвійний вихід, хтось лишився всередині). **Називай ці випадки сам** — це те, що оцінюють.

## Задача 9 — Кількість островів

**Що хочуть:** сітка з `'1'` (земля) і `'0'` (вода). Порахувати острови. Острів — це земля, з'єднана по горизонталі/вертикалі.

**Інтуїція:** уяви карту зверху. Ти бачиш клітинку землі — це новий острів. Тепер тобі треба "обійти" весь цей острів повністю (усю з'єднану землю), щоб не порахувати його шматки ще раз. Обходячи, ти "заливаєш" землю водою (міняєш `'1'` на `'0'`), наче зафарбовуєш уже пораховане.

**Що таке DFS (пошук углиб):** від клітинки йдеш до сусіда, від нього — до його сусіда, і так поки є земля. Це як розливати фарбу: вона тече по всій з'єднаній області.

**Покроково:**
1. Йдеш по всіх клітинках сітки.
2. Знайшов `'1'` → це новий острів, `count + 1`. Запускаєш "заливку" (`sink`).
3. `sink` рекурсивно йде в 4 сторони й міняє кожну `'1'` на `'0'`, поки не впреться у воду або край.
4. Далі продовжуєш обхід — уже залиту землю не порахуєш, бо вона стала `'0'`.

**Трасування** на сітці:
```
1 1 0
0 1 0
0 0 1
```
```
Зустріли (0,0)='1' → острів #1. Заливаємо: (0,0),(0,1),(1,1) — усі з'єднані → стають 0.
Сітка стала:
0 0 0
0 0 0
0 0 1
Зустріли (2,2)='1' → острів #2. Заливаємо (2,2).
Більше землі немає. Відповідь: 2
```

**Ключова ідея:** "заливка" гарантує, що кожен острів рахується рівно раз. Перевірка меж (`r<0||c<0||...`) — щоб не вийти за край сітки.

## Задача 10 — Групування анаграм

**Що хочуть:** згрупувати слова, що є анаграмами (з тих самих літер). `["eat","tea","tan","ate"]` → `[[eat,tea,ate],[tan]]`.

**Інтуїція — спільний "відбиток":** як зрозуміти, що `eat`, `tea`, `ate` — те саме? Якщо **відсортувати літери** кожного, усі дають `aet`. Це їхній спільний відбиток. Слова з однаковим відбитком — анаграми.

**Чому HashMap:** ключ — відсортований відбиток (`aet`), значення — список усіх слів із таким відбитком.

**Покроково:**
1. Береш слово, сортуєш його літери → ключ. (`eat` → `aet`)
2. Кладеш слово у список під цим ключем (створюєш список, якщо його ще нема).
3. У кінці значення мапи — це й є групи.

**Трасування** на `["eat","tea","tan","ate"]`:
```
eat → ключ "aet" → {aet: [eat]}
tea → ключ "aet" → {aet: [eat, tea]}
tan → ключ "ant" → {aet: [eat, tea], ant: [tan]}
ate → ключ "aet" → {aet: [eat, tea, ate], ant: [tan]}
Результат: [[eat, tea, ate], [tan]]
```

**Ключова ідея:** перетворюємо "складно порівняти слова" на "просто порівняти відбитки".

## Задача 11 — Перший унікальний символ

**Що хочуть:** знайти **перший** символ, що зустрічається лише раз. `"swiss"` → `'w'` (s повторюється, w — ні).

**Інтуїція:** два проходи. Спершу **порахувати** кожен символ. Потім пройти **в порядку рядка** й повернути перший із лічильником 1.

**Чому LinkedHashMap:** звичайна HashMap **не пам'ятає порядок**, у якому ти клав ключі. А нам треба "перший" — тобто порядок важливий. `LinkedHashMap` зберігає порядок вставки, тож другий прохід іде в правильній послідовності.

**Покроково:**
1. Перший прохід: рахуємо кожен символ у `LinkedHashMap`.
2. Другий прохід по мапі (у порядку появи): перший символ із кількістю 1 — відповідь.

**Трасування** на `"swiss"`:
```
Рахуємо: s→3, w→1, i→1   (порядок вставки: s, w, i)
Шукаємо першого з кількістю 1:
  s → 3, ні
  w → 1, ТАК → повертаємо 'w'
```

**Ключова ідея:** розділяємо задачу на "порахувати" і "знайти перший" — і `LinkedHashMap` дбає про порядок.

## Підсумок: як упізнати, що застосувати

| Сигнал у задачі | Інструмент |
|---|---|
| "порахувати скільки", "згрупувати", "знайти пару" | HashMap |
